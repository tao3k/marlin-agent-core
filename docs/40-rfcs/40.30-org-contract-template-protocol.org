#+title: 40.30 RFC: Org Contract Template Protocol

* 40.30 RFC: Org Contract Template Protocol

This RFC extends [[file:40.10-org-native-agent-work-protocol.org][40.10 Org-native Agent Work Protocol]] with parser-owned Org
contracts as template and validation protocol input.

** Motivation

Org templates should not be free-form strings owned by an agent prompt. They
need a parser-owned contract that can say which document or subtree shape is
expected, which Org elements must exist, and which message or fix template is
attached to a failed assertion.

=tao3k/orgize= now provides a =CONTRACT_ORG= contract mechanism. Marlin should
consume that mechanism directly rather than inventing a parallel template DSL.

** Scope

The protocol covers:

- projecting orgize contract registries into =marlin-org-model= DTOs;

- loading contracts beside workspace nodes through =marlin-org-workspace=;

- resolving document and subtree =CONTRACT_ORG= references against the local
  workspace and explicit contract documents;
  contract registry;

- exposing the projection through the =marlin-agent-core= facade;

- making the registry available to template engines and control planes as data.

The protocol executes only Marlin's parser-owned count/existence receipt subset
inside Marlin. It does not choose a template engine, does not load external
registry files from the filesystem yet, and does not give contracts direct
write authority.

** Contract Source

Contracts are ordinary Org subtrees:

- =CONTRACT_ID= identifies the reusable contract;

- =CONTRACT_SCOPE= defines whether it applies to a document or subtree;

- =CONTRACT_KIND= identifies the contract family;

- assertion children use =ASSERT_ID= and =SEVERITY=;

- =org-elements-query= and =org-elements-expect= blocks define parser-owned
  count-query assertions;
  assertions;

- =jinja2 :name message= and future fix blocks provide template-facing text.

** Native Crates

=marlin-org-model= owns stable DTOs such as =OrgContractRegistry=,
=OrgContract=, =OrgContractAssertion=, =OrgContractQuery=, and
=OrgContractTemplate=. State-like fields use typed wrappers instead of public
stringly state. Assertion expectation is a structured enum projected directly
from orgize contract facts: =Exists=, =NotExists=, =Count { op, expected }=, or
=Unsupported { label }=.

=marlin-org-workspace= owns projection from orgize contracts through
=OrgDocumentLoader::load_workspace=.

=OrgDocumentLoader::load_workspace= also returns =OrgContractResolutionReport=.
It resolves local contract ids and aliases, and records =ORG044= warning
diagnostics for empty or unresolved references.

=marlin-org-workspace= projects assertion =message= and =fix= blocks into
=OrgContractTemplate= records. The records preserve role, engine label, and
body text as data for template renderers or Gerbil control planes.

=OrgContractValidationReport= is the runtime-independent receipt surface for
contract assertion results. The workspace loader executes the parser-owned
count/existence subset during projection and records passed, failed, or skipped
assertion receipts without invoking an external runtime. Supported receipts
carry matched node ids. Skipped receipts carry typed skip reasons such as
unsupported expectations.

=OrgDocumentLoader::load_workspace_with_contracts= accepts explicit Org
contract documents and merges their registries before reference resolution.
=OrgDocumentLoader::discover_contract_documents= selects candidate documents
that actually contain parser-owned contract registries before callers pass them
into =load_workspace_with_contracts=. Filesystem walking remains caller-owned.
=MemoryOrgWorkspace::load_documents_with_discovered_contracts= owns the
workspace-level batch lifecycle for in-memory agents.

=marlin-workspace-view= and =marlin-workspace-status= expose contract facts as
selected view data and status summaries, including matched-node validation
evidence, structured expectation summaries, and typed skip-reason summaries,
without owning template execution. Both surfaces carry the typed
=OrgContractRegistry= and full validation report so rendered text is never the
only contract fact channel.

=marlin-gerbil-scheme= may pass =GerbilWorkspaceContractFacts= in
=GerbilCompileRequest= so command-backed Gerbil control planes can select
contract templates or emit patch intents from typed facts, including validation
skip reasons.
Workspace patch-intent compile requests carry default empty contract facts and
can be overridden with real parser-owned facts.

=marlin-agent-core= re-exports =org_model= and =org_workspace= so callers can
use contract registries from the facade without depending on parser internals.

** Mutation Boundary

Contracts are read-side facts. A Gerbil control plane or template engine may
consume them to select scaffolds, render messages, or produce
=WorkspacePatchIntentSpec=. Durable changes still go through =WorkspacePatch=,
=marlin-org-patch=, and =marlin-org-workflow=.

** Acceptance Criteria

- =orgize= is updated to a revision containing =CONTRACT_ORG= contracts.

- =OrgDocumentLoader::load_workspace= returns nodes and =OrgContractRegistry=.

- =OrgDocumentLoader::load_workspace= returns contract reference resolutions and
  =ORG044= diagnostics for empty or unresolved =CONTRACT_ORG= references.

- =OrgDocumentLoader::load_workspace= returns template-facing assertion
  =message= and =fix= facts as =OrgContractTemplate= records with role, engine,
  body, and source span facts.

- =OrgDocumentLoader::load_workspace= returns an =OrgContractValidationReport=
  for runtime-independent parser-owned assertion receipts during load, including
  typed skip reasons for skipped receipts.

- =OrgDocumentLoader::load_workspace_with_contracts= resolves references against
  explicit external Org contract documents.

- =OrgDocumentLoader::discover_contract_documents= filters candidate Org
  documents down to those that contain contract registries.

- =MemoryOrgWorkspace::load_documents_with_discovered_contracts= loads a batch
  of Org documents, discovers registry documents from that batch, and exposes
  merged contract facts in view and status.

- View and status DTOs can carry typed contract registries, diagnostics,
  validation records, and skip-reason summaries as typed fields, not only
  rendered summary counts for agent-facing context.

- Validation receipts carry matched node ids for supported parser-owned
  assertions and typed skip reasons for unsupported expectations.

- =GerbilCompileRequest= can carry contract registry, resolution, structured
  expectation, validation facts, and validation skip reasons through
  command-backed Gerbil adapters, and
  =GerbilCommandCompiler::compile_with_contract_facts= can send those facts
  through the JSON stdin command boundary.

- Workspace patch-intent compile requests include default contract facts in the
  normal =GerbilCommandCompiler::compile= path.

- The real-gxi workspace patch-intent workflow accepts command adapter requests
  carrying contract facts and still dry-runs and commits through the workflow
  receipts.

- Release landing reports carry observed evidence keys, observed artifact paths,
  and missing artifact paths so release/status artifacts can be audited without
  reading the full sidecar.

- Contract state fields are typed wrappers, and assertion expectation is a
  structured enum rather than a public string label.

- Unit tests parse a contract fixture through orgize and assert the Marlin
  registry, resolution, template, validation, view, status, and Gerbil request
  projections.

- Workspace gates pass after the dependency update.
