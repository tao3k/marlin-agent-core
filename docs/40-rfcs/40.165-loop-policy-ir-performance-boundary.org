#+DATE: 2026-06-23
#+STATUS: Implementing

* 40.165.00 Decision

Marlin will treat Gerbil/POO/C3-C4 as a semantic policy compiler and Rust/Tokio
as the execution kernel, state machine, and side-effect owner.

The hot runtime contract is:

#+begin_src text
POO roles / modules
  -> Gerbil semantic compiler
  -> LoopPolicyIR + LoopProgram + policy digest
  -> Rust GenericLoopMachine
  -> Model / tool / sandbox / session / workflow runtimes
  -> Typed receipts
#+end_src

Gerbil must not become an oracle that Rust asks on every token, tool, event, or
node.  A policy version is composed once, a run stage crosses the ABI once, and
a batch returns compact decisions.

* 40.165.10 Execution Layers

| Frequency | Work | Owner |
|-----------+------+-------|
| Config version change | POO combination, C3/C4, slot merge algebra, provenance | Gerbil |
| Run or iteration | Replan, failure analysis, graph rewrite, human gate decision | Resident Gerbil or compiled Scheme |
| Frontier or node batch | Admission, route, resource collision, budget decision | Rust compiled plan, optional batch native ABI |
| Token, tool, event | Stream, process, network, sandbox, cancellation, backpressure | Rust/Tokio |

* 40.165.20 Policy Compiler Requirements

** 40.165.20.10 Force

POO lazy slots are useful for incremental policy construction, but runtime
slots must be forced before the ABI handoff.  Required slots include
capabilities, budgets, gates, resource classes, route tables, continuation
tables, and graph templates.

** 40.165.20.20 Freeze

After compilation to a =policy_epoch=, the resolved policy pack is immutable.
Policy edits create a new epoch rather than mutating a running plan.

** 40.165.20.30 Slot Merge Algebra

C3/C4 decides role order.  Slot merge algebra decides meaning.

Required merge kinds:

- intersection for capability sets
- union for deny lists, human gates, and observability
- min for retry and cost ceilings
- ordered append for route rules
- conflict error for exclusive resource ownership

* 40.165.30 LoopPolicyIR v1

The first typed protocol is =ResolvedLoopPolicyPack= in
=marlin-agent-protocol=.  It has:

- =schema_version=
- =policy_epoch=
- =policy_digest=
- =hot=
- =audit=

The hot pack contains compact runtime data:

- capability and human gate masks
- budget caps
- compiled graph nodes and edges
- route index buckets
- resource classes
- defunctionalized continuation table
- maker and checker profile IDs

The audit pack contains cold evidence:

- slot provenance
- C3/C4 linearization
- diagnostics
- source locations
- explanation strings
- forced slots
- merge receipts

* 40.165.35 LoopProgram Boundary

The executable output of POO Flow is a provider-neutral =LoopProgram=. Marlin
must not add =CodexTurnDriver=, =ClaudeTurnDriver=, =OpenRathTurnDriver=, or
equivalent provider-branded runtime drivers.

The boundary is:

#+begin_src text
Loop mechanism policies
  -> POO Flow composition, inheritance, override, specialization
  -> Compiled LoopProgram
  -> Marlin GenericLoopMachine
  -> ModelInvocationRuntime / ToolDispatchRuntime / SandboxRuntime
  -> SessionRuntime / WorkflowRuntime / ReceiptRuntime
#+end_src

Provider adapters adapt only:

#+begin_src text
ModelRequest -> provider API -> ModelEvent / ToolRequest / FinalResponse
#+end_src

They do not own loop semantics, pending-input drain, retry policy, graph
rewrite, tool frontier selection, context compaction, approval gates, or stop
hooks.

Loop mechanism policies are named and composed in the POO layer. Initial
families include:

- =reactive-tool-loop-base=
- =pending-input-drain-policy=
- =tool-frontier-policy=
- =sampling-retry-policy=
- =context-compaction-policy=
- =stop-hook-policy=
- =approval-policy=
- =turn-diff-policy=
- =stream-event-policy=
- =resource-key-dispatch-policy=
- =dynamic-graph-rewrite-policy=
- =sub-agent-fanout-policy=
- =result-synthesis-policy=
- =adaptive-verification-policy=

The verification target is not provider imitation. The target is that the same
model and task can use different POO policy combinations and produce different,
explainable runtime behavior through typed =LoopProgram= transitions and
receipts.

Implementation evidence:

- =marlin-agent-protocol= defines =LoopProgram=, =LoopProgramInput=,
  =LoopMechanismPolicyId=, typed event kinds, typed action kinds, and
  defunctionalized transitions.
- =marlin-agent-kernel= exposes =GenericLoopMachine= as the provider-neutral
  execution state machine for =LoopProgram=. It emits typed
  =GenericLoopMachineReceipt= values and never calls provider APIs, tools, Tokio
  tasks, or Scheme during transition selection.
- =marlin-agent-kernel= exposes =LoopProgramRunRequest= and
  =LoopProgramRunReceipt= as the controller handoff surface. The handoff lives
  in a separate owner module from the graph-loop controller so =LoopProgram=
  execution does not inflate the existing =GraphLoopController= substrate.
- =marlin-agent-kernel= projects =LoopProgramActionKind= receipts into
  =LoopProgramRuntimeHandoffPlan= entries. Tool, memory, and placement actions

  handoff kinds rather than being disguised as tools.
- =marlin-agent-kernel= executes handoff plans through a typed
  =LoopProgramRuntimeHandoffExecutor=. The default executor emits =Deferred=

  fine-grained handlers for control, model, tool, memory, placement, runtime,

* 40.165.40 Defunctionalized Continuation

Scheme closures and continuation objects do not cross the runtime boundary.
They lower into typed operations:

- =StopCompleted=
- =StopFailed=
- =Retry=
- =Rewrite=
- =Defer=
- =Escalate=

Captured closure data must become explicit IDs, masks, counters, and template
references so Rust can persist, compare, replay, cancel, and recover after a
crash.

* 40.165.50 ABI Direction

The ABI is a delivery boundary for compiled strategy, not a per-event RPC
surface.

P0 keeps focus on eliminating repeated native selector conversions:

- separate match keys from payload
- keep payloads in Rust
- construct policy backing once per epoch
- compile prefixes into trie, DFA, or first-token buckets
- compile scopes into bitsets
- avoid rebuilding provider/model/name/context/isolation strings per call

P3 introduces batch evaluation:

#+begin_src c
int marlin_loop_evaluate_batch(
    uint64_t policy_epoch,
    const LoopFact *facts,
    uint32_t fact_count,
    LoopDecision *decisions);
#+end_src

Hot path requirements:

- no JSON
- no alist
- no per-call policy reconstruction
- no Scheme-owned output memory
- no callback
- no per-policy string copy

Rust builds full receipts from compact decision IDs plus the audit pack.

* 40.165.60 Resident Gerbil Lane

Resident Gerbil is reserved for:

- =ReplanRequired=
- =PolicyChanged=
- =UnknownFailureClass=
- =HumanDecisionArrived=
- =DynamicExtensionLoaded=

Normal node, tool, token, and event processing does not call back into Scheme.
Rust owns channels, backpressure, timeout, process health, cancellation, worker
restart, and load shedding.

Current bridge boundary:
- =GerbilResidentStrategyExecutor= is the Rust-owned execution trait behind
  accepted resident strategy requests.
- =GerbilResidentStrategyGxiSmokeBridge= calls the real
  =:marlin/deck-runtime-strategy= procedure
  =marlin-deck-runtime-resident-strategy-execute= and decodes its Scheme datum
  response into a typed Rust receipt.
- The bridge intentionally uses a Rust-owned Scheme datum codec for this
  explicit cross-process smoke boundary; it does not introduce JSON and does
  not make Scheme own runtime-internal text serialization.
- This bridge proves the procedure contract and typed receipt shape only. It is
  not a runtime policy executor.
- Native AOT is the core policy path. Runtime =.ss= policy under Rust must
  enter through a compiled native AOT handoff or an already-resident native
  transport, with millisecond to sub-millisecond dispatch as the required
  performance budget.
- Per-call =gxi= is retained only as a slow smoke baseline. The benchmark below
  is a rejection receipt for that transport, not an accepted performance result.
