* 40.140 Pi Agent Core Loop Parity Plan
:status: Active
:owner-crates: marlin-agent-protocol, marlin-agent-kernel, marlin-agent-runtime, marlin-agent-core, marlin-agent-harness, marlin-gerbil-scheme

** Purpose
The first graph-loop hardening target is functional parity with the loop behavior demonstrated by =.data/pi/packages/agent=. Marlin should learn the proven runtime shape, but not copy the transcript-first implementation model.

Marlin's loop remains a typed graph runtime:
- Rust owns protocol types, runtime handles, receipts, provider request projection, replay artifacts, CLI/debug surfaces, and execution envelopes.
- Gerbil Scheme owns policy composition over typed facts and continuation profiles.
- The provider boundary receives projected request messages; it does not own loop semantics, session semantics, or replay semantics.

** P0-P5 Implementation Plan
*** P0 Protocol And Receipt Surface
Define the minimum graph-loop parity protocol:
- =GraphLoopRunId=, =GraphLoopIterationId=, =GraphLoopEventId=, and =GraphNodeExecutionId= newtypes.
- =GraphLoopEvent= lifecycle taxonomy matching the parity matrix.
- =GraphLoopContinuationDecision= with receipt-bearing =accept=, =deny=, =defer=, and =rewrite= actions.
- =GraphLoopInputQueue= with steering/follow-up lanes and explicit drain policy.
- =GraphToolBatchExecutionReceipt=, =GraphToolCallReceipt=, =GraphToolBatchDecision=, and =GraphLoopStopReceipt=.

P0 is complete only when no-live-LLM protocol fixtures can serialize, replay, query, and reject invalid continuation requests without provider calls.

*** P1 Runtime Run Registry And Loop Handle
Add a runtime-owned run registry:
- start, cancel, wait, inspect, and complete handles;
- active run state with current iteration, pending node/tool ids, last event id, and terminal status;
- event stream tee that preserves capture subscribers even if primary output closes;
- cancellation and idle semantics that mirror pi's =abort= and =waitForIdle=, but return typed receipts.

P1 is complete when =marlin loop run=, =marlin loop inspect=, and a cancellation fixture share the same run id and replayable receipt path.

*** P2 Queue And Continuation Semantics
Implement steering/follow-up queues in the graph loop:
- steering drains before the next proposal/provider request;
- follow-up drains only after the loop would otherwise stop;
- drain-all and drain-one policies are explicit;
- assistant-tail continuation denial is a typed readiness receipt, not a thrown string.

P2 is complete when deterministic fixtures prove queue ordering across at least three turns.

*** P3 Tool Batch Execution Semantics
Lift the current debug process-command executor into a typed graph tool batch executor:
- sequential and parallel modes;
- per-tool forced sequential override;
- before/after hook receipts;
- bounded update events;
- source-order result artifacts even when parallel completion order differs;
- batch termination only when every finalized tool requests terminate.

P3 is complete when sequential, parallel, block, rewrite, error, update, and mixed-terminate fixtures all replay through =marlin graph query=.

*** P4 Session And Content Graph Integration
Map loop output into the project/session/content graph:
- content nodes for prompt, assistant partial/final, tool result, compaction, custom entry, branch summary, model change, tool activation change, and thinking-level change;
- session tree branch movement and labels;
- project/worktree/session/content provenance on every event and receipt.

P4 is complete when one fake user workspace can run, branch, compact, replay, and query without raw transcript sharing.

*** P5 Gerbil/POO Policy Plane
Expose the parity surface to Gerbil through native ABI projections:
- Gerbil POO profiles can decide continuation, stop, hook, model route, and queue policy over typed facts;
- Rust validates readiness and resolves handler catalogs;
- Scheme never owns JSON/protocol serialization and never constructs Rust handlers.

P5 is complete when a POO-driven no-live-LLM fixture chooses different hook/model/queue behavior for two agent classes and the replay artifact preserves typed Rust receipts.

** Non-goals
- Do not copy pi's transcript-first public API.
- Do not make the debug CLI the semantic owner of loop behavior.
- Do not add a Rust hook DSL for complex policy.
- Do not let Scheme own cross-process JSON or runtime handler construction.
