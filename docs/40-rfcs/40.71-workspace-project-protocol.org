* 40.71 RFC: Workspace Project Protocol

Status: active

This RFC defines the Marlin-native workspace project boundary learned from the
Codex workspace/projects design. The useful lesson is that project identity,
Git repository roots, VCS ecosystem extensions, trust, remote operations,
metadata, runtime import, and execution environment must be separate protocol
facts.

** Problem

For Marlin, a project is primarily a Git repository or worktree:

- the Rust-owned project boundary is the repository/worktree root and its Git
  metadata;

- Git operations are the core repository foundation in Rust;

- =jj= is an ecosystem VCS extension over that foundation, not an equal
  Rust-core backend;

- a stable project id names that repository-backed workspace across sessions
  and receipts;

- Org is not the project identity and does not define the repository boundary;

- Org files hold project/workspace metadata, memory, reviewer notes, and other
  agent-facing content;

- GitHub operations are attached to a typed remote repository such as
  =jj-vcs/jj=;

- complex =jj= policy is resolved by the Scheme extension plane and compiled
  back into typed Marlin project policy;

- runtime activation should import a typed project snapshot, not infer project
  identity from cwd or from an Org file.

** Codex Lesson

Codex is mature in treating workspace/projects as an explicit runtime and
state boundary. Marlin adopts the boundary discipline, not the product shape:

- project identity is a typed value;

- repository/worktree roots and Git metadata are explicit;

- Git operations are rooted at the imported Git top-level, not at ambient
  process cwd;

- ecosystem VCS choices, including =jj=, stay policy-driven extensions;

- trust is attached to paths before tools or sub-agents use them;

- GitHub issues, pull requests, checks, actions, and releases are typed
  operation families, not free-form shell commands;

- metadata files are separate from execution state;

- runtime receives a selected project instead of discovering one implicitly.

** Marlin Contract

=marlin-workspace-protocol= owns the durable project profile:

- =WorkspaceProjectId= is the stable project identifier;

- =WorkspaceProjectRepository= records the worktree root, Git core backend,
  optional Git metadata directory, VCS extensions, and trust gate;

- =WorkingCopyRepositoryDiscoveryPath= records the imported absolute path used
  as input to native Git top-level discovery;

- =WorkingCopyGitTopLevel= records the resolved absolute Git top-level path
  from the imported project's =.git= boundary;

- =WorkspaceProjectVcsBackend= anchors the Rust-owned core backend to =Git=;

- =WorkspaceProjectVcsExtension= records ecosystem VCS layers such as =jj=;

- =WorkspaceProjectPolicyPlane= records whether behavior is Rust-core or
  Scheme-extension policy;

- =WorkspaceProjectRoot= records additional roots that may belong to the
  project but do not define the repository identity;

- =WorkspaceProjectGitHubOps= records an optional GitHub remote, canonical
  =owner/repo= slug, and allowed operation families;

- =WorkspaceProjectPersistence= records repository-local state persistence;

- =WorkspaceProjectProfile= combines project id, repository boundary,
  additional roots, optional Org metadata file, and persistence policy;

- =WorkspaceCtx= may carry a project id so query, patch, view, validation, and
  status calls stay project-scoped without deriving identity from cwd.

=marlin-agent-protocol= may carry =RuntimeWorkspaceProject= as an imported
runtime snapshot for cwd, sandbox, activation, and config-layer selection. That
runtime snapshot is not the durable project profile.

** Org Metadata Boundary

Org metadata is a content file attached to the repository-backed project. It
may store workspace memory, reviewer evidence, contract notes, release state,
or agent metadata, but it must not replace the VCS repository boundary.

Markdown is not a compatibility surface for project metadata. If metadata needs
to be persisted for agent work, use Org-native files or typed Rust values that
project to Org.

** Runtime Boundary

Runtime crates consume typed project facts and emit receipts:

- active project import receipts record whether the requested project was
  imported;

- environment activation receipts record cwd, sandbox, activation plan, and
  delta;

- workspace protocol calls retain project identity through =WorkspaceCtx=.

Runtime must not infer trust from path shape, cwd, shell state, or Org file
presence. Trust gates come from typed project policy.

** P0 Working Copy Isolation

The near-term P0 is not a general VCS replacement layer. It is an agent working
copy isolation protocol:

- one agent can receive one isolated working copy path;

- the provider records whether isolation came from native =git worktree= or
  Worktrunk;

- GitHub pull request checkout is a typed operation family, not an opaque shell
  command;

- requests compile into provider-specific typed plan steps before runtime
  adapters execute commands;

- every planned, applied, or rejected isolation action emits a stable receipt.

=GitWorktree= is the Rust-core baseline provider. It covers create, switch,
list, and remove behavior over native Git worktree semantics. It intentionally
does not claim GitHub PR checkout because that requires a higher-level provider
or GitHub integration.

=Worktrunk= is the P0 agent-oriented provider. It builds on Git worktrees but
adds the workflow surface that matters for agent harnesses: branch-addressed
working copies, PR checkout, merge flow, hooks, and cleanup.

The protocol handoff to runtime is =WorkingCopyIsolationPlan=. A plan may
contain steps such as =GitWorktreeCreate= or =WorktrunkPullRequestCheckout=.
Runtime adapters may later translate those typed steps to concrete provider
commands, but shared protocol must not expose shell strings as the semantic API.

** P1 Command Projection

P1 adds a second handoff layer: =WorkingCopyCommandProjection=.

The projection turns typed plan steps into structured provider command
invocations:

- program is an enum such as =Git= or =Worktrunk=;

- =repository_discovery_path= is the imported absolute path used only as input
  to native Git top-level resolution;

- =git_toplevel= is the resolved Git top-level carried by command
  invocations, not an ambient process directory;

- args are argv entries, not a shell string;

- the expected working copy handle remains attached to the command.

This still does not execute commands. It gives runtime adapters a deterministic
projection target after a resolver converts discovery paths through native Git
top-level resolution. Provider command construction stays testable before
Tokio execution, cancellation, stdout/stderr capture, and receipts are added.

Examples:

- =GitWorktreeCreate= accepts a repository discovery path, resolves it through
  =git rev-parse --show-toplevel=, and projects to =git worktree add ...= with
  the resolved =git_toplevel=;

- =WorktrunkPullRequestCheckout= accepts a repository discovery path, resolves
  it through =git rev-parse --show-toplevel=, and projects to =wt switch
  --no-cd --no-hooks --format json pr:<number>= with the resolved
  =git_toplevel=.

The projection must reject requests that are under-specified for execution,
such as a Worktrunk pull request checkout without a repository discovery path.

Codex treats the caller process directory as a discovery/input detail, but
repo-scoped behavior resolves the repository root through =git rev-parse
--show-toplevel=. Marlin follows that split at the protocol layer: runtime
adapters resolve =repository_discovery_path= into =git_toplevel= through
=marlin-git-utils=, then use =git_toplevel= only as the provider process launch
directory. Public receipts and command projections name the Git semantic root
instead of the process detail.

** P2 Provider Executable Probe

P2 adds a runtime smoke boundary for provider executables.

The environment crate probes provider programs with non-mutating =--version=
commands before a real isolation driver has to modify a repository:

- =Git= probes the =git= executable;

- =Worktrunk= probes the =wt= executable;

- probe results record executable name, status, status code, stdout, and
  stderr;

- missing provider binaries are typed as =Missing= instead of being confused
  with provider command failure.

This keeps installation verification separate from working-copy creation. A
developer machine can run the real =wt= smoke once Worktrunk is installed
globally, while default tests remain portable.

=jj= workspaces remain a later VCS extension. They can provide parallel working
copies, but their value comes from change graph and rewrite policy. That policy
belongs in the Scheme extension plane and must compile back into typed Rust
receipts before runtime consumes it.

** P3 Runtime Driver

P3 connects protocol projection to runtime execution without duplicating provider
command semantics.

=marlin-agent-environment= owns the runtime driver:

- it compiles =WorkingCopyIsolationRequest= into =WorkingCopyIsolationPlan=;

- it resolves each =WorkingCopyRepositoryDiscoveryPath= through
  =marlin-git-utils= before projection;

- it calls =WorkingCopyCommandProjection::from_plan= after Git top-level
  resolution;

- it executes structured command invocations through an injectable runner;

- it emits =WorkingCopyCommandReceipt= values with status, stdout, stderr,
  status code, and expected working-copy metadata.

The runtime driver must not rebuild Worktrunk or Git argv independently.
Provider command construction is centralized in =marlin-workspace-protocol= so
the runtime boundary executes the typed projection instead of owning a second
command compiler.

** P4 Operation Family Coverage

P4 closes the initial working-copy operation family for Git and Worktrunk:

- =GitWorktree= supports create, switch, list, and remove through native Git
  worktree semantics;

- =Worktrunk= supports create/switch, list, remove, and GitHub pull request
  checkout through =wt=;

- remove behavior is represented by =WorkingCopyRemovalMode= instead of a bare
  boolean;

- switch creation behavior is represented by =WorkingCopySwitchMode= instead of
  a bare boolean;

- list visibility is represented by =WorkingCopyListOptions= instead of raw
  branch/remote flags;

- command receipts carry the expected working-copy handle when the provider
  operation is supposed to affect one.

The Worktrunk command projection is intentionally non-interactive for agent use:
=wt switch= uses =--no-cd --no-hooks --format json=, =wt list= uses
=--format json=, and =wt remove= uses =--foreground --no-hooks --format json=.

** P5 Scheme Extension Boundary

P5 keeps complex VCS and workspace policy outside the Rust command compiler
while preserving typed Rust receipts.

Rust owns:

- project identity;

- Git top-level discovery;

- provider executable probing;

- working-copy request, plan, command projection, and receipts;

- TOML configuration envelopes and outer CLI/trace serialization.

Scheme may decide:

- which provider should satisfy a policy;

- whether a =jj= extension should rewrite, defer, deny, or request a
  working-copy operation;

- how dynamic hook/customer-agent policy should select model, provider, and
  workspace behavior;

- which typed Rust request should be emitted for runtime execution.

Scheme must not own JSON serialization for this core path. JSON is allowed at
configuration, trace, CLI, or explicit cross-process protocol boundaries only.
Native internal integration is Scheme types -> Rust types -> typed
request/receipt values.

** GitHub And jj Operations

GitHub operations are project capabilities. They are not implied by the VCS
extension: a =jj= project may still have a GitHub remote, and a Git project may
choose not to enable GitHub operations.

The protocol should represent GitHub operation families explicitly:

- pull requests;

- issues;

- checks;

- actions;

- releases.

The =jj-vcs/jj= repository is the motivating shape: Git remains the Rust-core
operations foundation, =jj= is represented as a Scheme-policy VCS extension,
and GitHub remains the collaboration and operations plane.

** Acceptance Criteria

- =marlin-workspace-protocol= exports typed working-copy isolation providers,
  requests, operation families, and receipts for =GitWorktree= and =Worktrunk=.

- =WorkingCopyIsolationRequest= compiles into =WorkingCopyIsolationPlan= with
  provider-specific typed steps before any runtime adapter executes commands.

- =WorkingCopyIsolationPlan= projects into =WorkingCopyCommandProjection= only
  after repository discovery paths resolve to Git top-levels, with structured
  program, =git_toplevel=, argv, and expected working-copy metadata.

- =marlin-agent-environment= executes =WorkingCopyCommandProjection= through an
  injectable runtime driver and records command receipts without duplicating
  provider argv construction.

- =GitWorktree= and =Worktrunk= cover create/switch/list/remove where
  supported, and =Worktrunk= covers typed GitHub pull request checkout.

- Provider flags that affect behavior are represented by named protocol types
  such as =WorkingCopySwitchMode=, =WorkingCopyRemovalMode=, and
  =WorkingCopyListOptions=.

- =marlin-workspace-protocol= exports typed project id, Git core backend,
  repository, VCS extension, policy plane, root, trust, GitHub operations,
  persistence, and optional Org metadata contracts.

- Scheme policy may choose or rewrite typed working-copy requests, but core
  working-copy runtime integration remains Scheme types -> Rust types -> typed
  Rust requests/receipts. JSON is not a native-internal policy surface.

- =WorkspaceCtx= can carry project identity independently of runtime
  activation.

- Focused tests prove repository-first identity, Git core backend, =jj= as a
  Scheme-policy VCS extension, optional Org metadata, GitHub operation
  families, trust filtering, and project context.

- RFC 40.70 and 40.80 cite this RFC when applying Codex workspace/project
  lessons.
