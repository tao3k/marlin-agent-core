* 40.70 RFC: Codex Engineering Lessons Baseline

Status: active

This RFC records the first engineering baseline extracted from the Codex source
tree under =.data/codex=. This baseline is subordinate to
[[file:40.80-marlin-architecture-principles.org][40.80 Marlin Architecture Principles]].

The goal is not to clone Codex. Codex is treated as an engineering lessons
corpus: it helps identify mature problem boundaries, but it does not define
Marlin's implementation shape. Marlin keeps only the useful contracts and
replaces fixed product behavior with typed policy, Tokio-native execution,
ASP-owned evidence, and more configurable strategy surfaces.

** Source Baseline

The local Codex checkout is rooted at =.data/codex/codex-rs=. Its workspace is
large and intentionally split across protocol, core, app server, execution,
tools, hooks, model providers, state, rollout, observability, and configuration
crates.

One selector-level example from Codex is the app-server protocol boundary for
standalone process spawning. The method is protocol-owned and explicitly marked
experimental rather than being hidden in runtime internals:

- =process/spawn= owns argv-based process spawn parameters and response shape.
- =process/writeStdin= owns stdin transport for the spawned process.
- PTY-backed =command/exec= remains a separate protocol surface.

That lesson is worth adopting: capability entry points should be protocol facts
before they become runtime implementation details. The concrete Codex
implementation is not normative for Marlin.

Workspace and project management is another mature Codex boundary worth
learning from. Marlin records that narrower adaptation in
[[file:40.71-workspace-project-protocol.org][40.71 Workspace Project Protocol]]:
Git repository/worktree identity, jj-style VCS ecosystem extensions, explicit
roots, trust gates, GitHub operation families, optional Org-native metadata
files, runtime project import, and typed working-copy isolation stay separate
facts.

** Selector-scoped Isolation And Performance Audit
The project-scoped graph runtime work in [[file:40.120-project-scoped-agent-graph-runtime.org][40.120 Project-scoped Agent Graph Runtime]] must first pass through a narrower Codex audit focused on execution isolation, session envelopes, sandbox robustness, and output performance.

The current selector-scoped evidence from =.data/codex/codex-rs= shows these useful boundaries:

- =core/src/session/session.rs:46-108= keeps session-level provider, collaboration mode, approval policy, permission profile state, environment selections, workspace roots, =codex_home=, fork/parent thread ids, dynamic tools, inherited shell snapshot, and shell override in one session configuration envelope.
- =core/src/session/session.rs:436-457= warms plugin and skill state during session initialization using resolved environment selections, avoiding a cold capability lookup on the first turn.
- =core/src/config/resolved_permission_profile.rs:36-54= makes a resolved permission profile snapshot a bridge over already-resolved config state, not an ad hoc resolver.
- =core/src/sandboxing/mod.rs:42-61= defines a typed =ExecRequest= carrying command, cwd, env, network proxy, expiration, capture policy, sandbox type, Windows roots, permission profile, filesystem/network runtime policies, Windows overrides, and =arg0=.
- =core/src/sandboxing/mod.rs:158-171= routes execution through a small =execute_env= / =execute_exec_request_with_after_spawn= boundary rather than scattering process spawning through callers.
- =core/src/landlock.rs:14-70= launches the Linux sandbox helper with a canonical permission profile and lets the helper derive filesystem/network policy. Core does not embed Linux policy text directly.
- =linux-sandbox/src/landlock.rs:34-88= applies restrictions to the current thread so only the child inherits them, enables =PR_SET_NO_NEW_PRIVS= when needed, installs network seccomp when network is restricted, and keeps filesystem restrictions primarily in bubblewrap.
- =linux-sandbox/src/landlock.rs:119-163= keeps legacy Landlock filesystem rules as a fallback/reference path and treats non-enforced rulesets as sandbox errors.
- =linux-sandbox/src/landlock.rs:165-268= denies ptrace, process VM access, io_uring setup/enter/register, and network syscalls under restricted mode while allowing necessary process-local Unix socket behavior.
- =core/src/exec.rs:1-120= caps retained exec output bytes, caps live output delta events, and sets an I/O drain timeout so inherited stdout/stderr pipes from grandchildren cannot hang the agent indefinitely.
- =core/src/exec.rs:120-320= models timeout and cancellation as typed =ExecExpiration= variants and combines cancellation tokens without widening the process API.
- =core/src/exec.rs:320-780= transforms portable exec params into a sandbox-specific request and records Windows sandbox spawn failures as metrics rather than losing platform failure shape in text.
- =core/src/exec.rs:780-1440= explicitly maps timeout, sandbox-denied, and output aggregation behavior; it also admits that sandbox-denied classification is heuristic when backend receipts are not decisive.
- =core/src/exec.rs:1440-1660= drains stdout/stderr with bounded wait, aborts output tasks on drain timeout, streams at most =MAX_EXEC_OUTPUT_DELTAS_PER_CALL= deltas, and keeps reading to EOF to avoid pipe back-pressure while retaining only capped bytes.

Marlin should keep these lessons:

- Use typed execution envelopes. A graph/tool execution request should carry permissions, visibility, network policy, output capture policy, cancellation, working-copy provenance, and expected receipt shape as fields, not comments or shell text.
- Keep platform sandbox backends replaceable. Linux helper, macOS Seatbelt, Windows restricted/elevated backends, and future container backends should sit behind typed backend adapters with capability/readiness receipts.
- Apply sandbox policy in the narrowest inheritance boundary. If a restriction is process/thread inherited, the receipt must say so.
- Treat output volume and process lifetime as safety properties. Every tool path needs retained-output caps, live-stream caps, cancellation semantics, process-group cleanup, and drain timeouts.
- Prewarm capability metadata deliberately. Tool cards, skills, plugins, and environment selections should have a session-init warming path with receipts.
- Preserve session permission state atomically. Active permission profile id, concrete profile, workspace roots, and per-turn grants must update together.

Marlin should improve on these Codex weak spots:

- Replace heuristic sandbox-denied detection with backend-owned denial receipts wherever the backend can know the syscall, path, or network reason. Text inference should be marked as lower-trust fallback evidence.
- Avoid compatibility shims that preserve ambiguous behavior. Legacy sandbox projections may be migration aids, but old/new semantics must not stay active together.
- Avoid making a broad session configuration object the public contract. Split protocol-visible session scope, permission scope, content scope, tool scope, shell snapshot, and runtime config into typed sub-receipts.
- Keep platform-specific complexity out of the shared executor. Backend-specific validation, readiness checks, and failure receipts belong in backend modules.
- Keep product-specific fixed policy out of execution code. Policy should arrive through typed TOML and Gerbil/POO strategy projections, with Rust owning validation and execution.
** External Sandbox Backend Baseline
[[file:40.130-sandbox-backend-capability-protocol.org][40.130 Sandbox Backend Capability Protocol]] records the first external sandbox backend direction. The immediate lesson from Nono and CubeSandbox is that Marlin needs backend polymorphism, not a single baked-in sandbox.

Nono is useful evidence for a local developer backend: zero-setup local execution, macOS/Linux/WSL2 support, profile-based filesystem and network policy, and language bindings for adapter integration.

CubeSandbox is useful evidence for a server backend: RustVMM/KVM hardware isolation, E2B-compatible service APIs, high-density deployment, sub-60ms sandbox creation claims, x86_64 Linux/KVM/PVM readiness constraints, and snapshot/clone/rollback through CubeCoW.

Marlin should model both behind the same capability/readiness/receipt protocol. A backend may be excellent for one workload and invalid for another; that decision belongs in graph policy and typed readiness receipts, not in hard-coded executor branches.
** Mature Capabilities To Learn

- Protocol/schema ownership: public capabilities are named operations with
  parameter and response types.
- App server boundaries: client, daemon, protocol, and transport are separated.
- Execution boundaries: shell commands, sandboxing, escalation, and process
  hardening are not collapsed into agent logic.
- Tooling boundaries: tools, hooks, skills, plugins, and MCP integration are
  separate extension surfaces.
- Provider boundaries: model providers, backend clients, proxies, and realtime
  paths can evolve independently.
- State boundaries: thread store, rollout, memories, message history, and graph
  storage are independent persistence surfaces.
- Observability boundaries: rollout traces, debug context, telemetry, and
  event streams provide runtime evidence.
- Configuration boundaries: config, cloud config, auth, secrets, and feature
  flags are not owned by runtime task execution.

** Marlin Adaptation Rules

Marlin should adopt boundary discipline but keep the design independent,
smaller, more performant, and more replaceable:

- RFC first for new cross-crate capabilities.
- Shared protocol schema before runtime behavior.
- Runtime crates consume typed requests and policies, not raw TOML, Scheme, or
  Codex-specific config files.
- Provider and tool implementations stay replaceable behind thin facades.
- ASP parser-owned facts are first-class evidence, not after-the-fact logs.
- Sub-agent spawn policy stays declarative by default, with Scheme/Gerbil as an
  optional strategy layer after typed policy compilation.
- Public runtime APIs use named request/config objects instead of broad
  positional argument lists.

** Flexibility Improvements Over Codex

Codex has strong engineering boundaries, but many client-facing choices are
still fixed by the current product surface. Marlin should improve flexibility
in these areas:

- Configurable sub-agent profiles with explicit permissions, context visibility,
  and performance budgets.
- Strategy plugins that can rank or partition work without widening typed
  policy limits.
- Provider-native search and mutation frontiers that avoid raw source scanning.
- Hook decisions that produce reusable receipts and next commands.
- Runtime scheduling that is budget-aware, replay-safe, and cache-aware.
- Smaller crates with explicit ownership so a new capability does not require a
  monorepo-wide mental model.

** Baseline Workstreams

- Protocol baseline: define the operation, request, response, and receipt
  before implementation.
- Runtime baseline: map typed requests into Tokio tasks, cancellation,
  isolation, process observation, and event emission.
- Tool execution baseline: split process spawn, shell command, PTY command,
  stdin, sandbox, and escalation.
- Hook baseline: keep hook policy and client envelopes distinct from provider
  facts.
- State baseline: separate thread, memory, rollout, and trace persistence.
- Observability baseline: every high-risk runtime action emits stable evidence.
- Configuration baseline: TOML declares policy; Scheme/Gerbil may optimize
  strategy only after policy is typed.

** Steered Conversation Learning

The steered-conversation lesson from =.data/codex= is not a single runtime
object. Selector-scoped evidence shows a small typed turn payload:
=codex-rs/app-server-protocol/schema/typescript/v2/Turn.ts:9-33= models a
turn with =items=, =itemsView=, =status=, typed error, start/completion
timestamps, and duration. The ASP frontier for the same question also selected
=sdk/typescript/src/turnOptions.ts=, =v2/AdditionalContextEntry.ts=,
=v2/MemoryCitation.ts=, =ConversationSummary.ts=, and rollout-trace owners.

Keep these lessons:

- Conversation steering is per-turn context assembly, not full transcript
  sharing. The payload says which thread items are included now.
- Partial materialization is a first-class contract. A turn needs an item-view
  receipt so callers know whether they received a full, filtered, windowed, or
  otherwise bounded projection.
- Additional context, memory citations, and summaries should be typed protocol
  objects with source anchors. They should not be hidden in prompt text.
- Status, typed error, and timing belong beside the turn payload so replay,
  UI, and audit code can explain how a steering decision affected the turn.

Marlin should adapt the structure, not clone the product shape:

- Project-runtime turns should consume an explicit context pack assembled from
  Org graph recall, source spans, tool cards, and session/content facts.
- Memory shards should be referenced through citations and relationship facts,
  never sent as whole shard bodies.
- The landed steering receipt records selected item ids, omitted item reasons,
  source anchors, memory citation ids, and graph-query receipt ids while
  keeping raw transcripts and whole memory shards out of the protocol payload.
- The current 40.120/40.121 runtime line should treat this as the next
  conversation-layer contract after read-model evidence, before sandbox backend
  execution becomes the focus.

** Non-goals

- Do not vendor Codex code into Marlin.
- Do not mirror the full Codex workspace shape.
- Do not make runtime crates parse Codex config directly.
- Do not make Scheme/Gerbil the default user-facing config language.

** Acceptance Criteria

- Future runtime capabilities cite this baseline or a narrower RFC.
- New process/tool/sub-agent APIs expose named typed request objects.
- New configurable behavior has a declarative policy surface and tests.
- Codex source reads used for design decisions are selector-scoped and recorded
  as evidence.
- Implementation changes keep unrelated local dirty work isolated.
