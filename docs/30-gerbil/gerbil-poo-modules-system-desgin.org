#+title: Gerbil POO Modules System Design

* Gerbil POO Modules System Design
:PROPERTIES:
:ID: docs-30-gerbil-poo-modules-system-desgin
:KIND: gerbil_design
:STATUS: active
:END:

This document records the higher-level user model for Marlin's Gerbil POO
module system. The filename keeps the requested spelling, but the design term is
"modules system design".

The important distinction is that users should not assemble a module system from
zero. Upstream Marlin should provide a furnished policy surface: prefab modules,
policy packs, object catalogs, default imports, default profiles, typed options,
and projection receipts. Users then make small, explicit changes to that object
graph.

This complements [[file:30.90-poo-module-system.org][30.90 POO Module System]], which is the implementation landing
plan for the underlying =marlinModules=, =marlinEvalModules=, POO slot merge,
and typed projection machinery.
The milestone flow lives in [[file:30.91-poo-module-system-milestones.org][30.91 POO Module System Milestones]].

** Mental Model: Furnished House, Furniture, And Catalog

The core user-facing metaphor is important and should be preserved in every
implementation discussion:

#+begin_quote
Marlin should not ask users to pour concrete, lay wires, and build a house from
raw materials. Upstream Marlin should hand users a furnished house.
#+end_quote

The "house" is the upstream module pack. It already has rooms, wiring, plumbing,
doors, default furniture, and working appliances:

- rooms are module boundaries;
- wiring is the import graph;
- plumbing is the runtime/projection path;
- furniture is policy objects such as hook policies, model routes, matchers,
  subagent profiles, continuation profiles, scripts, and recovery strategies;
- appliances are Rust-owned runtime/catalog handlers that already exist;
- inspection reports are typed receipts.

The user should normally start from the furnished house:

#+begin_src scheme
(def workspace
  (CustomerAgentWorkspace
   (.o project-root: "."
       model-profile: "interactive")))
#+end_src

That should be enough to run. The user is not "writing a module system"; the
user is moving into an upstream prefab.

When the user wants a different chair, table, lamp, route policy, matcher, hook
selector, or subagent profile, the interface should let them replace that object
directly. They should not have to learn an option-priority game or search for a
hidden =disabledModules= key. Changing furniture should be object surgery:

#+begin_src scheme
(marlin-replace-object
 'model-route
 "default-model-route"
 my-model-route)
#+end_src

If a user really wants an unfurnished room, Marlin can also expose an
"unfinished shell" layer: low-level =marlinModules=, =marlinPolicyModule=,
=marlinPolicyExtension=, and explicit catalog/evaluation calls. That layer is
for framework authors and advanced users, not the default teaching interface.

This is the practical difference from simply saying "users write modules and the
system evaluates them." The upstream system supplies a whole object graph first.
The user edits that graph.

** Why This Is Not Just A Metaphor

The house/furniture model should shape the API:

- A prefab is a furnished house, not a loose bag of modules.
- A policy object is furniture with identity, family, owner, and receipt trail.
- A pack catalog is the furniture showroom and house catalog.
- Remove means removing a specific object from the house.
- Disable means leaving the object in the plan but marking it inactive.
- Replace means swapping one object for another object of the same family.
- Add means putting a new object into a named room/object family.
- A receipt is the inspection report that says what changed and who owns it.

This keeps the interface intuitive for users and precise for agents. A human can
say "replace the default review chair" while the system records
=family=hook-policy=, =target-id=default-human-review-hook=, and
=replacement-id=my-human-review-hook= in a typed receipt.

The "furniture" must not be untyped text. It should be a POO object. That is why
small lambdas are useful but should not stay naked: a matcher lambda, ranker
lambda, transformer lambda, or hook selector function should be wrapped as a
named policy object so receipts can track identity and Rust never has to parse
Scheme source.

** Goal

Marlin should expose a simple interface at the top, while preserving a precise
object graph underneath.

The desired user experience is:
The current maintained user-interface delivery path makes that even thinner: downstream authors define a small config object and call the upstream =UserInterface= entrypoint. The lower-level =UserInterfaceApply=, =UserInterfaceDeliveryReceipt=, and =UserInterfacePolicyProjection= functions remain available for debugging and framework authors, but ordinary downstream files should call one maintained upstream entrypoint.

#+begin_src scheme
(import :marlin/prefabs/customer-agent)

(def workspace
  (CustomerAgentWorkspace
   (.o project-root: "."
       model-profile: "interactive")))
#+end_src

That prefab should already contain the normal workspace, agent, hook, model
route, continuation, review, failure recovery, and catalog projection defaults.
The user can run it without learning the internal module plumbing.

When the user needs customization, they should extend or replace typed POO
objects:

#+begin_src scheme
(def workspace
  (.o (:: @ (list (CustomerAgentWorkspace default-customer-config)))
      (extensions => marlin-extensions-append
                  (marlin-extensions my-review-extension))
      (policy-objects => marlin-policy-objects-replace
                      (marlin-replace-object
                       'model-route
                       "default-model-route"
                       my-model-route))))
#+end_src

The point is object-level surgery, not option-priority tricks.

** Ownership Boundary

Rust remains the core execution engine:

- engine and runtime lifecycle;
- graph validation;
- typed ABI/projection validation;
- process/tool execution;
- receipts and replay;
- performance budgets;
- runtime catalog handler lookup.

Gerbil Scheme plus POO owns the policy/module control plane:

- module imports;
- option schema, config, default, and merge;
- policy composition;
- continuation profiles;
- subagent policies;
- hook selection policy;
- model routing policy;
- condition and matcher objects;
- module evaluation receipts.

Rust must not parse Gerbil source, implement a second module DSL, resolve user
imports, evaluate option merge semantics, or manufacture hook handlers from
Scheme text. Rust receives typed projections and receipt facts only after Scheme
has produced Scheme values.

** Four User Levels

These levels are easiest to explain with the house model. They are not just
implementation tiers; they are how the project should teach the API.

*** Level 0: Furnished Prefab

The user imports an upstream furnished module pack and supplies only a small
config object. This is the normal user path.

#+begin_src scheme
(def workspace
  (CustomerAgentWorkspace
   (.o project-root: "."
       agent-class: "customer-worker")))
#+end_src

This should feel like receiving the keys to a furnished house. The pack already
contains imports, schemas, defaults, extension objects, policy profiles, scripts,
and projection receipts.

The user should not need to know how the walls are framed. They should know
where the rooms are and which knobs are intended for normal use.

*** Level 1: Normal Configuration

The user overrides ordinary values without touching the object graph.

#+begin_src scheme
(def workspace
  (CustomerAgentWorkspace
   (.o model-profile: "review-heavy"
       hook-owner: "customer-worker")))
#+end_src

This maps to the current =marlin-module-interface= plus =marlinModules= pattern:
schemas live in the interface; user config is a small POO object.

In the house model, this is changing thermostat settings, choosing a model
profile, or setting the workspace root. The furniture layout stays the same.

*** Level 2: Object Surgery

The user adds, removes, disables, or replaces typed objects inside the prefab.
This is the main reason to use Gerbil POO instead of copying a Nix-like
=disabledModules= mechanism.

This level is where Gerbil POO is stronger than a classic module system. Users
can say "remove this chair", "replace this table", or "add this lamp" as object
operations. They should not have to encode those changes as priority fights.

Core operations should become first-class typed objects:

- =marlin-add-object= adds an object to a named object family.
- =marlin-remove-object= removes by typed identity.
- =marlin-disable-object= keeps identity and records a disabled receipt.
- =marlin-replace-object= replaces one object with another object of the same family.

Example:

#+begin_src scheme
(def workspace
  (.o (:: @ (list default-workspace))
      (policy-objects => marlin-policy-objects-remove
                      (marlin-remove-object
                       'hook-policy
                       "default-human-review-hook"))
      (policy-objects => marlin-policy-objects-replace
                      (marlin-replace-object
                       'failure-recovery-policy
                       "default-retry-policy"
                       my-retry-policy))))
#+end_src

The operation should produce a typed receipt:

#+begin_src text
object_surgery_receipt
  operation = replace
  family = failure-recovery-policy
  target_id = default-retry-policy
  replacement_id = my-retry-policy
  owner = gerbil-poo
  rust_handler_manufactured = false
#+end_src

Replace is often more important than remove. Many real customizations are not
"delete default behavior" but "use my default behavior instead":

#+begin_src scheme
(marlin-replace-object
 'human-review-policy
 "default-human-review-policy"
 customer-human-review-policy)
#+end_src

This is clearer than =mkForce= style priority: the receipt says which object was
replaced, by which replacement, in which family.

*** Level 3: Advanced POO Extension

Advanced users write new objects directly.

#+begin_src scheme
(defmarlin-policy-extension my-review-extension
  (source ":customer/review")
  (object
   (make-marlin-deck-runtime-subagent-policy-extension
    "my-review-extension"
    my-subagent-profile
    my-route-policy
    my-condition
    my-matcher
    my-dynamic-hook-action
    '((owner . "customer")))))
#+end_src

Small lambdas should not remain naked. Matchers, transformers, rankers, and
hook selectors should be wrapped as policy objects so receipts can name object
identity, source, owner, and projection target.

In the house model, this is not just moving furniture. It is building a custom
desk or installing a new appliance. The object can still enter the same catalog
and produce the same receipts, because it is a POO extension object rather than
untracked Scheme text.

** Prefab Catalog Model

Upstream should provide object catalogs, not scattered modules. A user should
not be handed a pile of boards, screws, and disconnected furniture. The catalog
should expose coherent prefab packs:

- a complete customer-agent workspace;
- a review-heavy workspace;
- a failure-recovery profile;
- a human-review profile;
- a memory-recall profile;
- a memory-trigger profile;
- a memory-retention profile;
- a memory-visibility profile;
- a model-routing profile;
- a context-compression profile.

Each catalog entry is a complete object graph with defaults. Users can take the
whole entry, configure it, or perform object surgery on it.

A policy pack is a complete object graph:

#+begin_src text
PolicyPack
  interface schema
  default config
  imports
  extension exports
  policy objects
  option merge strategy
  validation receipts
  native projection payload
  catalog ids
#+end_src

The public Scheme shape should be close to:

#+begin_src scheme
(def CustomerAgentWorkspace
  (marlinPolicyPack
   (.o id: "customer-agent-workspace"
       interface: CustomerAgentWorkspaceInterface
       defaults: customer-agent-defaults
       imports: customer-agent-imports
       policy-objects: customer-agent-policy-objects
       projections: customer-agent-projections)))
#+end_src

The current implementation already has the lower-level pieces:

- =marlinModules= creates POO module objects.
- =marlinPolicyExtension= and =defmarlin-policy-extension= mark .ss exports as
  module-managed POO extension objects.
- =marlinPolicyModule= and =defmarlin-policy-module= wrap policy modules.
- =marlinModuleCatalog= collects modules.
- =marlinEvalModules= returns scalar evaluation receipts.
- =marlinModuleSystemPresentation= returns the current complete user-facing
  presentation receipt.
- =marlinPolicyPack= and =marlinDefaultPolicyPack= package a furnished policy
  object graph.
- =marlinPolicyPackInventory= lists the furniture: family ids, object ids,
  disabled ids, hook ids, and surgery conflict counts.
- =marlinPolicyPackPresentation= turns the furnished pack into a stable native
  projection payload.
- =marlinPolicyProjection= wraps that payload in the fixed Scheme -> Rust
  =PolicyProjection<T>= envelope.

The real debug CLI now exposes this for the debug prefab path through
=marlin gerbil policy-receipt=. It can also load a user-authored =.ss= entrypoint
with =--entrypoint= and call a user-selected emission expression with =--call=,
while Rust still only decodes typed projection facts.

** PolicyProjection<T>

The fixed handoff model is:

#+begin_src text
Scheme module output:
  PolicyProjection<T>

Rust receives:
  T = typed protocol struct
  receipt = validation/evaluation receipt
  catalog ids = existing Rust handlers only
#+end_src

The key boundary is that Rust does not know how the policy was merged. Rust only
knows whether the typed projection is valid, whether the handler ids exist,
whether the budget is acceptable, whether the graph is valid, and whether the
receipt is replayable.

=marlinPolicyProjection= is the Gerbil-side envelope for this contract. It names:

- =module_evaluation_receipt_kind=;
- =policy_projection_receipt_kind=;
- =native_projection_payload_kind=;
- =budget_receipt_owner=;
- =catalog_resolution_receipt_owner=;
- =import_graph_owner=;
- =option_merge_owner=;
- =extension_composition_owner=;
- =policy_composition_owner=;
- =runtime_lifecycle_owner=.

In the furnished-house model, =marlinPolicyPackPresentation= is the inspection
report for the house and its furniture, while =marlinPolicyProjection= is the
sealed folder handed to Rust. Rust can inspect the folder, validate typed facts,
and look up existing catalog handlers. It still cannot parse Scheme source or
manufacture handlers from Scheme text.

** Required Receipts

Every module pack evaluation should expose the same receipt pattern:

#+begin_src text
module_evaluation_receipt
policy_projection_receipt
native_projection_payload
budget_receipt
catalog_resolution_receipt
object_surgery_receipt
module_system_presentation_receipt
#+end_src

The current debug CLI already proves the presentation boundary with scalar
facts such as:

- =module_system_presentation_kind=;
- =module_system_projection_chain_kind=;
- =module_system_import_graph_owner=;
- =module_system_option_merge_owner=;
- =module_system_extension_composition_owner=;
- =module_system_native_projection_payload_owner=;
- =module_system_catalog_resolution_receipt_owner=;
- =module_system_rust_parses_scheme_source=;
- =module_system_scheme_manufactures_rust_handlers=.

The prefab layer should add object surgery facts without changing Rust into a
module evaluator.

** Nix And Grafonnix Lessons

Nix is still an important reference because its user surface is successful:
users learn =imports=, =options=, and =config=, then build large systems from a
small vocabulary. Marlin should keep that success. The top-level interface
should be easy enough that a user can say "use the customer-agent prefab, change
these values" without learning Marlin internals.

Marlin should not copy Nix's harder escape hatches as-is. In Nix, disabling a
module, replacing a default, or forcing a value often routes through
=disabledModules=, =mkForce=, =mkOverride=, priorities, module keys, path keys,
and ambiguity checks. That is understandable for Nix, but it is the wrong mental
model for Gerbil POO.

In the house model, Nix often makes users find the hidden breaker box to disable
a lamp. Marlin should let the user point at the lamp object and remove, disable,
or replace it. The operation is not "win a priority fight"; the operation is
"replace this typed object with that typed object".

Grafonnix points at the better shape for Marlin: upstream functions return
extendable POP objects, and users customize the returned object through
extension methods. The lesson is not about Grafana dashboards. The lesson is the
interface shape:

#+begin_src text
upstream returns object -> user extends object -> object remains typed
#+end_src

For Marlin, the returned object is a furnished policy pack, not a dashboard row.
The extension methods operate on policy objects, module objects, catalog entries,
and projection receipts. This is how Marlin gets Nix-level usability without
Nix-style escape-hatch complexity.

** Implementation Map

Current implementation:

- =crates/marlin-gerbil-scheme/gerbil/src/modules/lib.ss=
  owns the public facade for =marlinModules=, =marlinModuleCatalog=,
  =marlinEvalModules=, =marlinModuleSystemPresentation=, prefab packs, policy
  object surgery, and =marlinPolicyProjection=.
- =crates/marlin-gerbil-scheme/gerbil/src/modules/policy-pack.ss=
  owns the furnished pack constructor, default pack, pack catalog, inventory,
  presentation payload, and projection envelope.
- =crates/marlin-gerbil-scheme/gerbil/src/modules/*-policy.ss=
  owns the upstream policy furniture families for workspace, session, agent,
  hook selection, model route, continuation profile, human review, evidence
  graph, failure recovery, memory trigger, and catalog projection. The default
  pack now calls those family constructors instead of embedding anonymous
  objects.
- =crates/marlin-gerbil-scheme/gerbil/src/marlin/deck-runtime-debug-policy-extension.ss=
  owns the debug policy extension object and debug policy module presentation.
- =crates/marlin-gerbil-scheme/gerbil/src/marlin/deck-runtime-policy-receipt-gate-cli.ss=
  is the Scheme-owned debug entrypoint that emits scalar facts for Rust.
- =crates/marlin-gerbil-scheme/examples/user-interface-module-config/=
  is the maintained downstream example that now exposes catalog, evalModules,
  presentation, workflow, policy extension object, script, and LoopGraph
  continuation surfaces.
- =crates/marlin-agent-core/src/debug_cli/gerbil_cmd.rs= launches the Scheme
  entrypoint and decodes facts; it does not generate Scheme source.

** Next Implementation Slice

The prefab layer, debug prefab projection path, family policy constructors, and
user-authored =.ss= entrypoint loader are now present. The next code slice
should make the loaded file carry a complete custom policy-pack fact set
instead of wrapping the debug prefab emission:

- a fixture/example where the loaded =.ss= defines and emits its own pack id,
  object ids, operation receipts, projection chain, owner split, and timing;
- a performance receipt that separates process startup, file load, module
  evaluation, projection construction, and policy loop time;
- Rust typed structs for the loaded =PolicyProjection<T>= facts after the
  custom-pack receipt shape is stable;
- A maintained prefab example, likely
  =examples/user-interface-policy-pack=, that starts from an upstream pack and
  demonstrates Level 0 through Level 3 customization using family constructors
  for model route, hook selection, continuation profile, human review, and
  memory trigger overrides.

The acceptance gate should include:

- direct =gxi= tests for pack construction and object surgery;
- the existing =user-interface-module-config= example;
- =marlin gerbil policy-receipt= showing the pack presentation and
  =policy_projection_*= facts through the real debug CLI, including the
  =--entrypoint= / =--call= path;
- =asp gerbil-scheme check --changed .= and full provider check.
* Current User-Facing Module-System Checkpoint
This checkpoint pins the current design intent for the 1-6 landing slice.

The downstream user interface is a furnished prefab, not a raw module-system construction exercise. The normal user writes a small POO config object and calls the upstream =UserInterface= entrypoint. The apply, delivery receipt, projection, catalog, and evaluation helpers stay available as debug/framework surfaces.

Memory policy is a furnished object family pack. Recall, trigger, retention, and visibility are separate POO policy families so users can replace the chair they mean to replace. Rust still owns memory graph reads, snapshots, session visibility validation, persistence, budget checks, typed projection validation, and replay receipts.

The user-facing rule stays simple:
- upstream Marlin ships the furnished house;
- user .ss files configure or perform typed object surgery;
- policy furniture is named by family/id/owner metadata;
- Scheme/POO owns composition and option merge;
- Rust receives typed projections and receipts only.
