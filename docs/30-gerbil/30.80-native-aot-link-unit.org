* 30.80 Native AOT Link Unit

The native AOT path for =marlin-deck-runtime= is a Gerbil/Gambit build
boundary, not a default Cargo build-script side effect.

The Scheme crate owns the Gerbil package assets, the C ABI types, and the typed
Rust plan for producing a native link unit. The crate does not link that unit
from its own =build.rs=. Runtime or real LLM integration crates must opt in to
linking the native unit so normal Scheme package builds, tests, and docs do not
inherit native compiler and Gambit linker costs.

** Artifact Shape

The native AOT build has four ordered compiler phases:

1. =gxc -target C -s -S -O= emits the generated Gambit Scheme file.

2. =gsc -target C -obj= compiles the generated Scheme module object.

3. =gsc -target C -link= emits the non-flat Gambit link C source.

4. =gsc -target C -cc-options -D___LIBRARY -obj= compiles the link C source.

The link source must stay non-flat. =gsc -link -flat= avoids emitting =main=,
but it also drags Gambit base globals into the link object and collides with
=libgambit.a=. The upstream Gambit embedding shape is ordinary =gsc -link= plus
=-D___LIBRARY= when compiling the generated link C source.

The Rust static-link plan must consume both objects:

- module object: =deck-runtime-native~0.o=

- link object: =deck-runtime-native~0_.o=

Linking only =deck-runtime-native~0.o= is incomplete. It misses the Gambit
link-unit symbols that initialize the generated module set.

The required native ABI exports are:

- =marlin_deck_runtime_initialize=

- =marlin_deck_runtime_select_model_route=

The initializer is part of the hot ABI contract. It calls Gambit =___setup= with
=setup_params.version = ___VERSION= and the generated linkfile linker
=___LNK_deck_2d_runtime_2d_native_7e_0__=. Rust linked consumers must initialize
the Gambit runtime before calling the selector. Omitting this produces a
linkable binary that crashes or fails at the first selector call.

** Cargo Boundary

=GerbilDeckRuntimeNativeAotConfig::build_link_unit= is the Rust entrypoint for
producing the link unit. =GerbilDeckRuntimeNativeAotBuildReceipt::static_link_plan=
projects a successful build into Cargo linker directives.

=marlin-deck-runtime-native= owns the opt-in Cargo integration. Its
=linked-native= feature runs a build script that emits the two generated object
files and the Gambit link directives. The default crate build remains pure Rust
and does not invoke =gxc=, =gsc=, a C compiler, or the Gambit linker.

The feature build accepts these environment overrides:

- =MARLIN_DECK_RUNTIME_NATIVE_C_COMPILER=

- =MARLIN_DECK_RUNTIME_NATIVE_GAMBIT_LINK_LIBRARY=

- =MARLIN_DECK_RUNTIME_NATIVE_GAMBIT_LINK_SEARCH_DIR=

When =MARLIN_DECK_RUNTIME_NATIVE_GAMBIT_LINK_SEARCH_DIR= is absent, the build
support crate derives candidate =lib= directories from =default_gerbil_gsc_program=
and the canonicalized =gsc= path. It accepts =libgambit.a=, =libgambit.dylib=,
=libgambit.so=, and versioned Linux =libgambit.so.*= files. Explicit environment
configuration remains authoritative.

The build-support crate uses Rust bindings helpers only around the native
boundary; they do not replace the Gerbil/Gambit compiler pipeline. =cc= selects
the host C compiler through the standard Cargo environment contract and packages
the two =gsc=-produced objects into one static archive for linked consumers.
=cbindgen= is used as a generated-header audit surface. =object= verifies
required ABI symbols from real object files before falling back to shell =nm=.
=static_assertions= and =memoffset= gate the Rust =repr(C)= ABI shape in tests.
The native AOT build receipt records this audit path through
=GerbilDeckRuntimeNativeSymbolAuditMethod= so harnesses can distinguish the
object-file parser path from the shell symbol-table fallback.
Both audit paths inspect the module object and the generated Gambit link object,
because the native link unit is not complete unless both objects are consumed.

The helper CLI prints explicit metadata for downstream consumers:

- =MARLIN_GERBIL_NATIVE_AOT_MODULE_OBJECT=

- =MARLIN_GERBIL_NATIVE_AOT_LINK_SOURCE=

- =MARLIN_GERBIL_NATIVE_AOT_LINK_OBJECT=

- =MARLIN_GERBIL_NATIVE_AOT_GAMBIT_LINK_LIB=

- =MARLIN_GERBIL_NATIVE_AOT_GAMBIT_LINK_SEARCH= when configured

Downstream crates decide when those values become =cargo:rustc-link-arg=,
=cargo:rustc-link-search=, and =cargo:rustc-link-lib= directives.

** Performance Baseline

The current real native AOT benchmark is gated by
=MARLIN_GERBIL_NATIVE_AOT_BENCH= so default benches do not repeatedly compile
Gerbil/Gambit artifacts.

The current complete link-unit build result is:

#+begin_src text
linked-native --no-run: passed; generated AOT objects linked into the Rust test
binary with required symbols marlin_deck_runtime_initialize and
marlin_deck_runtime_select_model_route.
#+end_src

This number is the build-time cost for a complete native link unit. It is not
the runtime selector cost.

The native selector ABI is a typed C boundary. It returns
=MarlinDeckRuntimeModelRouteSelection= with =matched= and =policy_index= fields.
It must not serialize JSON or allocate runtime-owned receipt bytes on the hot
path. JSON is reserved for configuration, tracing, CLI receipts, and
cross-process protocol boundaries.

The Scheme package follows the same split. Core modules such as
=:marlin/deck-runtime= and =:marlin/deck-runtime-strategy= now expose Scheme
values and POO objects through =marlin-deck-runtime-model-route-selection= and
=marlin-deck-runtime-strategy-selection=. They do not emit text receipts.
Process adapters such as =:marlin/deck-runtime-policy= remain the place where a
CLI or cross-process boundary can translate those values for Rust-owned outer
protocol handling.

The long-term native conversion path is:

Rust validates the C layout before projecting it into
=GerbilDeckRuntimeModelRouteSelectionReceipt=. The stable selector status codes
are =OK=0, =NULL_POINTER=2, =ABI_MISMATCH=3, and =INVALID_SELECTION=4=. An
unmatched selection must use =MARLIN_DECK_RUNTIME_NATIVE_NO_POLICY_INDEX=.
Matched selections must use an index into the request policy array.

The runtime-facing bridge consumes that typed policy index through
=CompiledModelRouteResolver::resolve_selected_policy_index=. This keeps the
projection Rust-native and independent of the Gerbil crate: the linked native
selector chooses an original policy index, runtime revalidates that selected
policy against the current =ModelRouteRequest=, and the result is the same
=ModelRouteDecision= used by the ordinary resolver. The real LiteLLM stream
gateway only consumes the decision endpoint and does not need to link Gerbil or
know about the native ABI.

The dedicated =marlin-deck-runtime-native= crate owns this composition boundary.
It depends on =marlin-gerbil-scheme= and =marlin-agent-runtime=, while
=marlin-agent-runtime= and =marlin-agent-stream= remain free of Gerbil native
linkage. The crate feeds a priority-ordered native policy list to the Scheme
selector, maps the returned native-list index back to the original route rule
index, and then calls the runtime projection API. Unsupported native policy
predicates are treated as selector candidates only when they can be represented
as Deck command prefixes and agent-scope labels; Rust projection remains the
final correctness gate.

selector directly through =select_policy_index=:

The previous JSON-returning smoke path measured the wrong boundary because it

Complex Deck strategy policies live on the Gerbil Scheme extension plane rather
than the native selector ABI. The current strategy process benchmark is gated by
=MARLIN_GERBIL_REAL_STRATEGY_BENCH= and runs the package-shipped
=:marlin/deck-runtime-strategy= module through a real =gxi= script process:

#+begin_src sh
MARLIN_GERBIL_REAL_STRATEGY_BENCH=1 \
  cargo bench -p marlin-gerbil-scheme --bench deck_runtime_native -- \
  deck_runtime_real_scheme_strategy
#+end_src

On 2026-06-13 the local benchmark for
=deck_runtime_real_scheme_strategy/complex_strategy_process_roundtrip= was
approximately 563-636 milliseconds. This measures the full =gxi= process
roundtrip for session, agent-lineage, workspace-state, Org-memory, customer
agent, and dynamic hook-action strategy selection. It is the correct boundary
for policy-plane expressiveness checks, not for hot model-route selection. The
microsecond native ABI target remains scoped to the linked selector path above.

Dynamic hook action application is measured on the Rust runtime side, not
through the LLM gateway or a =gxi= process. The current benchmark command is:

#+begin_src sh
direnv exec . cargo bench --locked -p marlin-agent-hooks --bench dynamic_actions -- hook_dynamic_actions_register_rewrite_dispatch
#+end_src

On 2026-06-13 the local
=hook_dynamic_actions_register_rewrite_dispatch= benchmark measured
approximately 18.4-23.0 microseconds for catalog-backed =Register= plus
=Rewrite= dispatch. Criterion reported this run as slower than the previous
local sample, so the stable regression threshold remains open. This is the hot
runtime-action budget; Scheme policy evaluation and process adapter costs remain
separate measurements.

The compiled macro policy path is the middle performance tier between process
strategy execution and native C ABI selectors. The
=:marlin/deck-runtime-compiled-policy= module exposes
=defmarlin-deck-runtime-compiled-route-selector=, which expands stable policy
templates into direct Scheme selector functions at compile time. This keeps the
policy shape in Gerbil Scheme while removing the runtime rule-object
interpretation loop for policies that are stable enough to compile with
=build.ss= but not yet simple enough to push into the native selector ABI.

The current benchmark gate is feature-disabled by default:

#+begin_src sh
MARLIN_GERBIL_REAL_COMPILED_POLICY_BENCH=1 \
  direnv exec . cargo bench -p marlin-gerbil-scheme --bench deck_runtime_native -- \
  deck_runtime_real_scheme_compiled_policy
#+end_src

This benchmark writes the crate-shipped Gerbil package, runs =build.ss compile=
so the macro module is compiled, then executes a script that imports the
compiled policy module and calls the macro-generated selector. Its performance
should be compared with =deck_runtime_real_scheme_strategy= before promoting a
strategy template into native ABI code.

On 2026-06-13 the local
=deck_runtime_real_scheme_compiled_policy/compiled_policy_process_roundtrip=
benchmark measured approximately 428.73-495.93 milliseconds with a 460.52
millisecond midpoint. This is a real =gxi= process roundtrip gate, not the
steady-state selector hot path. Use it to verify that the compiled macro module
can be package-built and executed from Rust-owned assets; use a resident runtime
or native ABI benchmark for hot-path promotion decisions.

The same run measured
=deck_runtime_real_scheme_compiled_policy/compiled_policy_batch_10000_process_roundtrip=
at approximately 438.61-535.60 milliseconds with a 484.07 millisecond midpoint.
That outer Criterion sample still starts a fresh =gxi= process per iteration,
so the accepted hot-path evidence is the internal loop emitted by the compiled
policy module itself. The policy-object selector loop reported
=policy_elapsed_us= of =3988= for 10,000 selector calls, about 398 nanoseconds
per call. The direct policy-index selector loop reported =index_elapsed_us= of
=2188= for 10,000 selector calls, about 218 nanoseconds per call. This is the
current Scheme compiled-macro promotion gate: complex policy shape remains in
Gerbil Scheme, while the hot selector can avoid object projection and runtime
rule interpretation.

A later same-day rerun of the linked native ABI selector provides the current
hot-path control sample for this audit:

- =deck_runtime_linked_native_selector/1_policy=: 1.5715-1.9232 microseconds.
- =deck_runtime_linked_native_selector/8_policy=: 9.3581-12.816 microseconds.

Criterion reported both rerun samples as slower than the previous local
baseline recorded earlier in this section, so this path still needs regression
work. Even with that regression, it is the right order of magnitude for the
runtime selector path; the =gxi= process benchmarks above are not.

The native AOT command runner infers the Gerbil/Gambit prefix from =gxc= and
=gsc=. When the prefix has =bin=, =lib/gerbil=, =include/gambit.h=, and a
Gambit runtime library, the runner injects =GERBIL_HOME= and =GAMBOPT= for the
compiler subprocesses. This avoids repairing a source-built Gerbil tree just to
make Gambit resolve =~~bin=, =~~lib=, and =~~include=.

The next implementation step is to run the linked selector benchmark under a
Linux CI image and compare policy-count scaling before optimizing the Scheme
selector loop. The native hot path remains typed C ABI selection; JSON receipts
are allowed only at trace or cross-process protocol boundaries, outside the
native selector path.
