#+title: 20.60 Org Contracts And Templates

* 20.60 Org Contracts And Templates

Org contracts are parser-owned template and validation facts. They are authored
as ordinary Org subtrees and parsed by =tao3k/orgize= through its
=CONTRACT_ORG= contract mechanism.

** Contract Shape

The first supported shape mirrors orgize contracts:

- contract headings carry =CONTRACT_ID=, =CONTRACT_SCOPE=, and =CONTRACT_KIND=;

- assertion headings carry =ASSERT_ID= and =SEVERITY=;

- =org-elements-query= source blocks select required document facts;

- =org-elements-expect= source blocks describe expected counts or existence;

- =jinja2 :name message= and future fix blocks provide template-facing text.

** Native Projection

=marlin-org-workspace= exposes =OrgDocumentLoader::load_workspace=. It returns
both the projected =OrgNode= list and an =OrgContractRegistry= projected from
orgize contract facts.
=OrgDocumentLoader::load_workspace_with_contracts= accepts explicit Org
contract documents and merges their contracts into the registry before
reference resolution.

=marlin-org-model= owns the stable protocol DTOs. It keeps contract scope, kind,
severity, expectation, element category, element kind, and relative scope behind
typed wrapper boundaries so the agent protocol does not expose stringly state.

** Reference Resolution

The workspace loader also projects =CONTRACT_ORG= references into
=OrgContractResolutionReport=. Document-level keyword references and subtree
property references are parsed with orgize =parse_contract_reference= and
resolved against the local =OrgContractRegistry= by contract id or alias.

Unresolved or empty references produce =ORG044= warning diagnostics with the
parser-owned source span for the reference.

** Template Boundary

Contracts are template inputs, not direct mutations. =marlin-org-workspace=
projects assertion =message= and =fix= blocks into =OrgContractTemplate=
records with role, engine label, and body text. A template engine or Gerbil
control plane may consume those records to choose document scaffolds, messages,
and fix intent. Durable mutation still goes through =WorkspacePatch=,
=marlin-org-patch=, and =marlin-org-workflow=.

** Validation Receipts

=OrgContractValidationReport= is the runtime-independent validation result
container. It carries assertion receipts and diagnostics as data.
=marlin-org-workspace= executes a narrow parser-owned first pass while loading
a document: resolved =CONTRACT_ORG= references produce receipts for count-based
assertions over projected node summaries and properties. Unsupported
expectations remain explicit skipped receipts rather than invoking an external
runtime.

** View And Status

=marlin-workspace-view= exposes =ContractFacts= as a selectable compact view
field and carries rendered contract summary lines beside node text.
=marlin-workspace-status= has a =ContractStatus= summary for resolved
references, unresolved references, diagnostics, templates, validation receipts,
validation outcomes, and rendered summary lines.

** Current Limits

The first projection preserves orgize query, expectation, message, fix,
template facts, source block spans, local reference resolution, explicit
external contract documents, and runtime-independent validation receipts for
the count-query subset. It does not discover registry files implicitly and does
not execute template engines inside Marlin. Richer assertion execution remains
an upstream orgize lint concern until Marlin expands the receipt protocol beyond
the current parser-owned count summary path.
